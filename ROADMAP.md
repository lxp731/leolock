# 🗺️ LeoLock 项目路线图 (ROADMAP)

LeoLock 致力于成为 Linux/Unix 环境下最安全、最易用、且高性能的命令行加密工具。本路线图概述了项目从 v1.0 到 v2.0 的演进路径。

## 🎯 核心愿景
- **绝对安全**: 内存零残留，算法前沿，元数据防篡改。
- **极致性能**: 支持超大文件流式处理，多核并行处理。
- **无感体验**: 直观的 CLI 交互，完善的 Shell 集成。

---

## 🟢 阶段 1: 核心加固与性能优化 (v1.1.0) - [✅ 已完成]
本阶段重点解决了架构在处理大文件和敏感数据内存安全方面的瓶颈。

- [x] **流式 I/O 重构 (High Priority)**
    - 实现了基于 1MB 分块的 `BufReader/BufWriter` 流式处理。
    - 成功解决 GB 级大文件处理性能，实测 3GB 文件约 14 秒。
- [x] **内存安全强化**
    - 全面集成 `zeroize`，确保密码和密钥在内存中即时擦除。
    - 引入 `Zeroizing<T>` 包装器，利用 RAII 机制保证清理。
- [x] **原子化文件操作**
    - 实现了 `.tmp` 临时文件写入与 `fs::rename` 原子替换逻辑，防止数据损坏。
- [x] **元数据完整性保护 (AAD)**
    - 升级 V3 文件格式，将头部元数据纳入 AES-GCM 附加认证数据 (AAD) 保护。

## 🟡 阶段 2: 并行化与高级特性 (v1.2.0) - [🚀 下一阶段目标]
本阶段旨在提升批量处理效率并丰富功能维度。

- [ ] **多线程并行加密**
    - 使用 `rayon` 库实现目录递归时的并行处理，充分利用多核 CPU 性能。
- [ ] **元数据填充 (Padding)**
    - 对加密后的文件名元数据进行对齐填充，消除因文件名长度泄露的潜在隐私风险。
- [ ] **高级密码策略**
    - 支持从环境变量、Keyring (系统钥匙串) 或 stdin 加载密码。
    - 支持可选的“密码强度实时评估”反馈。
- [ ] **交互增强**
    - 引入 `indicatif` 库，为大文件和批量操作提供精美的进度条。

## 🟠 阶段 3: 配置灵活性与兼容性 (v1.5.0)
本阶段关注工具的可定制性和长期稳定性。

- [ ] **动态算法参数配置**
    - 允许用户在 `config.toml` 中自定义 Argon2id 的内存和迭代参数。
    - 算法参数随文件头存储，确保参数调整后的向后兼容性。
- [ ] **多格式输出支持**
    - `list` 命令支持导出为 JSON, CSV 格式，方便与其他工具集成。
- [ ] **FUSE 挂载支持 (实验性)**
    - 探索将加密目录挂载为虚拟文件系统，实现实时解密读取。

## 🔴 阶段 4: 生态系统与扩展 (v2.0.0)
本阶段标志着项目进入成熟期，侧重于平台扩展和企业级特性。

- [ ] **多语言 SDK**
    - 提供 C 绑定，支持 Python/Node.js 等语言调用 LeoLock 核心加密逻辑。
- [ ] **硬件密钥支持**
    - 集成 PKCS#11 接口，支持使用 YubiKey 等硬件令牌存储主密钥。
- [ ] **云端密钥库集成**
    - 支持从 HashiCorp Vault 或 AWS KMS 获取/备份加密盐值 and 配置。

## 🛠️ 架构设计原则 (Technical Principles)

1.  **Fail-Safe (故障安全)**: 任何文件操作必须假设会中途崩溃，数据完整性永远高于删除便利性。
2.  **No Secret Left Behind (零秘密残留)**: 内存中的敏感信息生命周期应尽可能短。
3.  **Explicit over Implicit (显式优于隐式)**: 所有的加密模式转换（如 Fast 模式）必须在日志中有清晰的提示。
4.  **Forward Compatibility (向前兼容)**: 文件头设计必须预留足够的版本扩展空间。
